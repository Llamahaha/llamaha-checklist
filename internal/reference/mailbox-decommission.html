Procedure

Mailbox decommission procedure

Step-by-step for properly offboarding a user's Exchange Online mailbox: sign-in block, forwarding, retention / hold, shared mailbox conversion, license handling, and the specific scenarios where Defender for Office 365 Plan 1 (Defender P1) still needs to ride along on the ex-user's mail. Use this for every termination ticket and flag license changes in the client CRM when done.

Intake

Confirm the request before you touch anything

Confirm with HR / requester

Termination effective date and time, whether this is amicable or hostile (changes your timeline), forwarding target (manager, delegate, or none), whether there is a legal hold request, and whether the user had any shared / delegated / admin access that needs transfer.

Check for a hold flag

If HR or Legal has flagged a hold, do not remove the license or delete the account until a Litigation Hold or Microsoft Purview eDiscovery hold is in place. Deleting a licensed mailbox without a hold starts the soft-delete clock (30 days) and you lose recoverability after that.

Confirm the Defender question up front

Before you remove anything, ask: will mail still flow through this mailbox after termination (forwarding, shared mailbox, delegated access)? If yes, Defender P1 coverage probably needs to stay attached to the resulting recipient. See the Defender P1 section below.

Ticket fields to capture

User display name, UPN, primary SMTP, sAMAccountName, termination date, forwarding target UPN, hold status, device serial(s) in their name, Teams / SharePoint / OneDrive ownership flags, and any shared mailboxes they own rather than just access.

Day 0

Block sign-in and stop the clock

Block sign-in (don't delete)

Entra admin center > Users > the user > Block sign-in. Or PowerShell: Update-MgUser -UserId user@tenant.com -AccountEnabled:$false. Never delete the Entra account on Day 0 unless there is no mail, files, or identity tied to it — deletion starts the 30-day recovery window and cascades to the mailbox.

Revoke active sessions

Entra > Users > user > Revoke sessions. Or PowerShell: Revoke-MgUserSignInSession -UserId user@tenant.com. This forces reauthentication across all Microsoft-connected apps, including Outlook, Teams, mobile, and browser sessions. Critical for hostile terminations.

Reset password

Reset to a long random string and do not store it. This is belt-and-suspenders behind the sign-in block — covers you if the block propagation lags or if a cached token is still valid somewhere.

Wipe MFA methods

Entra > the user > Authentication methods > delete each registered method (Authenticator, phone, FIDO key). Prevents a bad actor from completing MFA if they somehow still have the password.

Retire / wipe mobile devices

Intune > Devices > find devices owned by the user > Retire for company data removal (BYOD) or Wipe for full reset (corporate). Confirm with HR whether the company laptop is coming back or being reimaged in place.

Mail Flow

Out of office, forwarding, and delegation

Set the Out of Office

Exchange admin center > recipients > mailboxes > user > Others > Automatic replies. Internal and external versions, typically pointing new senders to the manager or a shared resource mailbox. Keep the tone neutral — the OOO is the last customer-visible thing the departed user "says."

Set forwarding (if requested)

Exchange admin center > the mailbox > Mailbox > Manage email forwarding. Prefer mailbox-level forwarding (survives profile changes) over an inbox rule (often lost when mailbox is converted). PowerShell: Set-Mailbox user@tenant.com -ForwardingSmtpAddress manager@tenant.com -DeliverToMailboxAndForward $true.

External forwarding caveat

If the forwarding target is outside the tenant, verify the org's outbound anti-spam policy allows automatic external forwarding. Defender's default is to block it. If it's blocked and the client still needs it, document the exception request before toggling it — auto-forward to external is a common data-exfiltration path and you want an audit trail.

Transfer delegated access

If the user had Full Access or Send As on other mailboxes, those permissions don't auto-cleanup. Run Get-Mailbox | Get-MailboxPermission -User user@tenant.com and Get-RecipientPermission -Trustee user@tenant.com to find them. Remove or reassign.

Shared mailboxes they owned

If the departed user was the sole Full Access user on a shared mailbox, the shared mailbox is now orphaned. Add the manager or a continuity account before Day 30 or nobody can open it.

Retention & Hold

Decide how long the mailbox lives

No hold, standard offboarding

Tenant retention policies apply to the mailbox content. After the retention window, the content ages out. Safe to convert to shared mailbox once forwarding is stable (typically 30–90 days post-termination per the client's SOP).

Litigation Hold

Exchange admin center > mailbox > Others > Manage litigation hold. Or PowerShell: Set-Mailbox user@tenant.com -LitigationHoldEnabled $true -LitigationHoldDuration Unlimited. Litigation Hold requires Exchange Online Plan 2 (included in E3, E5, and Business Premium; standalone or in Business Basic/Standard it is not included).

Purview eDiscovery hold

For matter-specific retention with legal oversight, use Microsoft Purview > eDiscovery (Standard or Premium) > create case > add custodian > apply hold. Premium eDiscovery adds custodian management and workflow and requires E5 / E5 Compliance add-on.

Inactive mailbox

The clean path for long-term retention without a license: place the mailbox on hold (Litigation or eDiscovery), then remove the license or delete the user. The mailbox becomes "inactive" — it is preserved indefinitely at no license cost and is searchable via Purview content search. Removing the license before applying the hold deletes the mailbox at the 30-day soft-delete mark.

Order matters

Hold FIRST, then license change. Repeat: hold FIRST, then license change. The most common offboarding mistake is removing the license immediately after termination "to save money" on an account that turns out to need legal preservation.

License Handling

Convert to shared, or keep licensed, or go inactive

Convert to shared mailbox

Exchange admin center > the mailbox > Convert to shared mailbox. Or PowerShell: Set-Mailbox user@tenant.com -Type Shared. Requirements: the mailbox must be under 50 GB (else keep Plan 2 license), and the source user account must still exist (even if blocked) until conversion completes.

Shared mailbox license rules

A shared mailbox under 50 GB with no archive, no hold, and no Defender-scoped policies does NOT need a license. License is required when: mailbox > 50 GB, online archive enabled (EOL Plan 2 or EOL Archiving add-on), Litigation Hold enabled (EOL Plan 2), or Defender for Office 365 P1 policies apply to that recipient (see Defender section).

Remove the license — after hold, after conversion

Microsoft 365 admin center > user > Licenses and apps > uncheck the plan. Or PowerShell with Set-MgUserLicense. Do this only after: (1) any required hold is applied, (2) mailbox converted to shared (if going the shared route), and (3) the Defender P1 decision is made.

Inactive mailbox path

If no one needs ongoing access and you just need to preserve: apply hold, remove license, delete the Entra user. Mailbox becomes inactive and is preserved indefinitely with no seat cost. Recoverable later if needed.

Keep licensed (avoid when possible)

Valid when the mailbox is >50 GB and can't be trimmed quickly, or when the client explicitly wants business continuity with full mailbox features. Flag the ongoing cost to the client and set a calendar reminder to revisit in 90 days.

Defender P1

When Defender for Office 365 Plan 1 is still needed

What Defender P1 covers

Safe Attachments, Safe Links, anti-phishing with impersonation protection, and real-time detection signals for email and Teams. It is per-recipient licensed — policies evaluate against licensed recipients in the tenant. Included in M365 Business Premium, A5 / E5 Security, and sold standalone around $2/user/month.

Scenario A — Shared mailbox still receiving external mail

Most common case. You converted the ex-user to a shared mailbox for continuity, external mail still flows in, and the tenant has Defender policies. Whether the shared mailbox needs its own Defender P1 depends on how the policies are scoped: tenant-wide Preset Security Policies require the recipient to be licensed to enforce; custom policies scoped to specific recipients follow the scope. Keep Defender P1 on the shared mailbox if anyone is reading from it and the client relies on Safe Attachments / Safe Links coverage.

Scenario B — Auto-forwarding to an internal user

If mail forwards to the manager inside the tenant and the manager has Defender P1, coverage is already in place at the destination. The decommissioned mailbox itself usually doesn't need P1 if it's acting purely as a transport hop. But — if the client wants evaluation to happen at the original recipient (for reporting / threat explorer attribution under the ex-user), keep P1 until forwarding stops.

Scenario C — Auto-forwarding to an external address

Defender's external-forwarding block usually fires first, so you've likely already exempted this flow. Keep P1 on the forwarding recipient while external forwarding is active — Safe Attachments / Safe Links still evaluate the content before it leaves the tenant, and you want that protection on mail routing to an external inbox.

Scenario D — Inactive mailbox with eDiscovery hold

Defender P1 is not strictly required to maintain an inactive mailbox — that's an Exchange Online Plan 2 / hold license question, not a Defender one. However, if the client's compliance program requires Defender audit and threat signals to persist for the user's mailbox history, keep P1 licensed until the mailbox is placed inactive. Once inactive, no new mail arrives, so ongoing P1 coverage is moot.

Scenario E — Delegated / transferred to a still-active user

Permissions handed to a live licensed user whose own license already includes Defender P1 (Business Premium, E5 Security). No additional P1 needed on the source mailbox. Document the delegate so the next tech sees the handoff.

Cost framing for the client

Defender P1 standalone is roughly $2/user/month. Compared to the cost of one phishing incident reaching a shared mailbox that the manager reads daily, that's nothing. When in doubt and mail is still flowing, keep P1 on and note it in the ticket close.

How to verify coverage is active

Microsoft 365 Defender portal > Email & collaboration > Policies & rules > Threat policies. Check that the relevant policy (Preset or custom) has the shared mailbox / forwarding recipient in scope. Run a Safe Links test URL through the mailbox and confirm detonation in Threat Explorer.

Associated Data

OneDrive, Teams, SharePoint, and groups

OneDrive handoff

SharePoint admin center > More features > User profiles > Manage user profiles > find the user > Manage site collection owners > add the manager as Primary Admin. Or set a secondary owner via PowerShell (Set-SPOUser). OneDrive retention is driven by the account deletion timeline (default 30 days, extendable to 10 years via tenant setting).

Teams cleanup

If the user owned any Teams, transfer ownership to another member before the account is disabled or deleted (a Team with no owner goes orphaned). Remove the user from Teams they were a member of last, after chat handoff is complete.

SharePoint site ownership

Check whether they were listed as a site owner on any SharePoint site. Reassign via SharePoint admin center > Active sites > select site > Permissions.

Distribution lists and security groups

Remove from mail-enabled DLs. Security groups last — sometimes you need the user's group membership context to investigate odd access later. Document what you removed in the ticket.

Third-party M365 backup

SkyKick, Dropsuite, Veeam M365, Afi — confirm the final backup ran after the last mailbox change (OOO, forwarding, conversion) and retention is set per client SLA. Deleting the user in M365 often removes them from the backup tool's billing the following cycle, but not always — check.

Timeline

Standard decommission schedule

Day 0 — Termination effective

Block sign-in, revoke sessions, reset password, wipe MFA, retire devices, set OOO, configure forwarding, apply hold if requested, inventory delegated access.

Days 1–30 — Stabilization window

Monitor forwarding for issues, respond to delegate handoff questions, confirm manager can access what they need, confirm Defender policies still apply where expected.

Day 30 — License review

If no hold and OOO has been running a month, convert to shared mailbox and remove the user license. If hold is in place, remove license and go inactive. Keep Defender P1 if mail still flows per scenarios A/C above.

Day 90 — Forwarding sunset

Turn off forwarding and OOO unless the client explicitly asked for longer. After 90 days, senders have either learned the new contact or the mail wasn't urgent.

Day 180+ — Final cleanup

If nobody has accessed the shared mailbox in 60 days and there is no hold, delete per client policy. If held, leave as inactive mailbox indefinitely. Remove from remaining groups and any stale permission assignments.

Ticket Close

Before you close the ticket

Confirm with HR

Short email summarizing what was done: sign-in blocked, sessions revoked, forwarding to X, hold status, license changes, devices retired. Saves a follow-up ticket two weeks later.

Update the client CRM

Record license changes so monthly billing reflects reality. Note Defender P1 kept / removed so the next reviewer doesn't second-guess the decision. Flag the calendar reminder for Day 90 sunset.

Document the Defender decision

In the ticket notes: which scenario (A–E above) applied and why P1 was kept or removed. This is the detail that comes up in the next offboarding audit.

Escalation checkpoints

If anything in this procedure was skipped (usually: the hold step), flag it to the lead. If the client requested something outside the standard flow (e.g., export to PST for the user personally), confirm authorization in writing before acting.

Quick Reference

Decision matrix

Legal hold requested

Apply Litigation Hold (needs EOL Plan 2) or Purview eDiscovery hold FIRST → remove license → mailbox goes inactive → indefinite preservation at no seat cost.

No hold, mail still flowing

Set OOO + forwarding → after 30 days convert to shared → remove user license → keep Defender P1 on the shared mailbox if mail still arrives and the client uses Defender.

No hold, clean break

Block sign-in → short OOO window → after 30–60 days delete the user → mailbox deleted at 30-day soft-delete mark → no ongoing license cost.

Large mailbox (>50 GB)

Cannot convert to shared as-is. Either archive / trim first, or keep licensed until retention ages content below 50 GB, or place on hold and go inactive.

Defender P1 quick rule

Mail still flowing to a recipient a human reads? Keep P1. Mailbox inactive or deleted? Drop P1. In doubt? Keep P1 — $2/month beats one phishing incident.
